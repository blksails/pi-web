# Requirements Document

## Project Description (Input)
attachment 存储后端可插拔化 + UnionBlobStore 多后端组合。依据 docs/attachment-union-store-design.md(pre-spec 设计稿)的 Spec 1 范围:registry 提为端口、BlobStore.put 回执化并把选中后端固化进描述符、新增 UnionBlobStore 组合多个具名后端、config 工厂支持 PI_WEB_ATTACHMENT_BACKENDS JSON env 拓扑、spawn env 透传使主/子进程重建同构组合。硬约束:后端绑定持久化在描述符层、配置完全 env 可表达、门面「先落 blob 再写描述符+失败回滚」不变式不动。S3 型后端(registry 与 blob 双实现)含在范围内。

## Introduction

现状附件系统的字节存储与描述符注册表均硬编码本地文件系统,唯一后端实现落盘于单机目录:云端多副本部署下附件不可共享、服务重建后字节丢失,附件能力事实上把部署形态钉死在单机。本 spec 使附件存储后端经配置选择、支持多个具名后端并存(含 S3 兼容对象存储),并保证「附件在哪个后端」随附件本身持久化——附件的分发生命周期长于会话与进程,历史消息中的附件在会话结束、服务重启后必须仍可分发。存量单机部署在不做任何配置变更时行为完全不变。

## Boundary Context

- **In scope**:后端可插拔化(字节与描述符两层)、多具名后端并存与组合、S3 兼容对象存储后端、运维经环境变量声明拓扑、主/子进程按同一配置各自重建一致的存储视图、存量对象(无后端绑定)的读取回退。
- **Out of scope**:agent 定义声明附件 profile(后续 spec `agent-attachment-profile`);存量附件的后端绑定批量回填工具(迁移期由读取回退覆盖);按会话枚举的性能重构(语义正确性在范围内,吞吐优化不在);附件配额/生命周期管理。
- **Adjacent expectations**:签名分发、会话属主、`origin` 语义、上传/工具产物落库的既有行为契约由既有 `attachment-store` / `attachment-tool-bridge` spec 拥有,本 spec 不改变其对外语义;协议描述符形状变更仅限新增可选字段,存量消费方无感。

## Requirements

### Requirement 1: 存量部署零行为变化

**Objective:** As a 存量单机部署的运维者, I want 不做任何配置变更时附件系统行为与升级前完全一致, so that 升级不构成迁移事件。

#### Acceptance Criteria
1. While 未设置多后端拓扑环境变量, the 附件存储服务 shall 以既有单一本地文件系统后端提供全部附件能力(上传、工具产物落库、签名分发、按会话枚举、本地路径解析),落盘目录与目录缺省规则与现状一致。
2. While 未设置多后端拓扑环境变量, when 附件落库, the 附件存储服务 shall 产出与现状形状一致的附件描述符(不含后端绑定字段亦为合法形状)。
3. The 协议层 shall 将后端绑定表达为附件描述符上的可选字段,使不携带该字段的存量描述符与存量消费方在校验与渲染上均不受影响。

### Requirement 2: 多后端拓扑经环境变量声明

**Objective:** As a 宿主运维者, I want 用单一环境变量声明具名后端集合、默认写入目标与描述符注册表位置, so that 部署拓扑可版本化、可复制,且不需要改代码。

#### Acceptance Criteria
1. When 设置了多后端拓扑环境变量且内容合法, the 附件存储服务 shall 按声明实例化全部具名后端并以声明的默认写入目标承接新附件。
2. If 拓扑声明不合法(JSON 不可解析、后端重名、后端集合为空、默认写入目标未在集合中声明、后端种类未知), then the 附件存储服务 shall 在装配期以明确指出错误项的报错失败,不得以部分拓扑静默启动。
3. The 拓扑声明 shall 以间接引用宿主环境变量名的方式表达远端后端凭据,拓扑声明体本身不含明文凭据。
4. If 拓扑声明引用的凭据环境变量在进程环境中缺失, then the 附件存储服务 shall 在装配期报错并指出缺失的变量名。

### Requirement 3: 写路径后端选择与绑定持久化

**Objective:** As a 宿主运维者, I want 每个新附件落库时选定一个具名后端并把该绑定随附件描述符持久化, so that 附件的可读性不依赖任何运行期状态。

#### Acceptance Criteria
1. When 新附件落库且多后端拓扑生效, the 附件存储服务 shall 将字节写入按写入策略选定的单一具名后端,并将该后端名固化进附件描述符。
2. The 附件存储服务 shall 保持「先落字节、再写描述符;描述符写入失败时回滚已落字节且不返回引用」的既有不变式,在多后端拓扑下语义不变。
3. When 附件描述符已固化后端绑定, the 附件存储服务 shall 在后续读取、签名分发、删除时仅依据该绑定路由,不依赖会话或进程的存活状态。

### Requirement 4: 读路径路由与存量回退

**Objective:** As a 终端用户, I want 历史会话中的附件在服务重启、会话结束后仍可正常显示, so that 会话历史完整可回放。

#### Acceptance Criteria
1. When 按 id 读取或签名分发一个携带后端绑定的附件, the 附件存储服务 shall 从绑定指向的后端取字节,行为与单后端场景一致(含未找到 → 既有类型化未找到错误)。
2. When 按 id 读取一个不携带后端绑定的附件(存量对象), the 附件存储服务 shall 按拓扑声明顺序逐后端探测,命中即返回;全部未命中报既有类型化未找到错误。
3. If 附件描述符绑定的后端名未在当前拓扑中声明, then the 附件存储服务 shall 报明确指出该后端名的配置错误,不得静默降级为探测。
4. When 服务进程重启后收到对既有附件的签名分发请求且签名有效, the 附件存储服务 shall 正常返回字节,无论该附件所属会话是否仍存在。

### Requirement 5: S3 兼容对象存储后端

**Objective:** As a 云端部署的运维者, I want 字节与描述符都能落 S3 兼容对象存储, so that 多副本实例共享同一附件视图且实例重建不丢数据。

#### Acceptance Criteria
1. Where 拓扑声明了 S3 兼容后端, the 附件存储服务 shall 支持将附件字节写入并读出该对象存储,包括流式读取与元信息查询。
2. Where 描述符注册表声明为 S3 兼容存储, the 附件存储服务 shall 将附件描述符持久化到该对象存储,且按 id 读取、按会话枚举的语义与本地注册表一致。
3. When 两个服务实例配置同一拓扑且其一落库附件, the 另一实例 shall 能按 id 读取该附件的描述符与字节。
4. Where S3 兼容后端承载某附件, when 为其签发分发地址, the 附件存储服务 shall 产出调用方可直接访问的有时效地址,时效语义与既有签名分发一致。

### Requirement 6: 主/子进程存储视图一致

**Objective:** As a agent 工具作者, I want 工具在子进程落库的附件与主进程分发的附件是同一份, so that 工具产物引用在前端始终可渲染。

#### Acceptance Criteria
1. When 主进程装配会话子进程且多后端拓扑生效, the 会话装配层 shall 向子进程完整下发重建同一拓扑所需的全部环境变量(含拓扑声明与其引用的凭据变量)。
2. When 子进程按下发的环境变量重建存储视图并落库工具产物, the 主进程 shall 能按返回的附件 id 完成签名分发与描述符读取。
3. If 子进程环境中未下发附件存储配置, then the 子进程附件能力 shall 维持既有降级语义(能力不可用的类型化安全拒绝,不崩溃子进程)。

### Requirement 7: 删除语义

**Objective:** As a 宿主运维者, I want 删除在多后端下保持幂等且作用于正确的后端, so that 清理操作可安全重试。

#### Acceptance Criteria
1. When 删除一个携带后端绑定的附件, the 附件存储服务 shall 仅在绑定后端上执行删除,且对不存在的对象幂等(不抛错)。
2. When 删除一个不携带后端绑定的附件, the 附件存储服务 shall 对拓扑内全部后端执行幂等删除。
