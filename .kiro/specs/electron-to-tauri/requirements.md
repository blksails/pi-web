# Requirements Document

## Introduction

把 pi-web 桌面壳从 Electron **原地替换**为 Tauri v2，并随包携带 Node 二进制作为 sidecar，以保留「系统未安装 Node 也能运行」这一既有保证。

桌面壳不含业务逻辑，只做四件事：**拉起本地后端**、**开窗加载本地回环 UI**、**提供两项原生能力（目录选择、外链打开）**、**退出时收尾进程树**。本次迁移的目标是**行为等价**：除本文档显式列出的差异外，替换前后对用户与运维者可观察的行为应保持一致。

### 迁移动机（唯一）

用户明确指认两项成本，构成本次迁移的全部动机：

1. **内存与启动开销** —— Chromium 常驻进程使空闲内存与冷启动时间高于必要水平。
2. **包体积** —— Electron 运行时本身约 100MB 量级。

更细粒度的权限安全模型、Rust 生态与未来原生能力等常被引用的理由**不作为本 spec 的目标**，不得据此扩大范围。

> ⚠ **动机内在张力**：随包携带 Node 二进制会向包体加回 40~60MB，因此「包体积」动机能否兑现是**未经验证的假设**。Requirement 11 将其转为可测的验收口径：必须给出替换前后的实测对比，若净收益不显著，交由用户重新裁定是否值得。同理，「内存与启动开销」必须有实测基线，不接受「新方案理论上更轻」的论证。

### 可行性前置风险（design 阶段必须以 PoC 消解）

窗口加载的是**远端 http 回环页面**而非随包静态页。已知上游开放缺陷（tauri-apps/tauri#11934）表明：此种加载方式下，宿主注入渲染层的全局桥**可能不生效**，从而使「渲染层调用宿主命令」的通路不可用。这直接威胁 Requirement 6 所依赖的实现手段（但不改变 Requirement 6 的可观察行为——退路是让目录选择改由后端端点承接）。design 阶段必须先以最小 PoC 验证该通路，再据以选定手段。

## Boundary Context

- **In scope**：桌面壳的进程编排、窗口加载、原生目录选择、外链处理、启动失败呈现、退出收尾、随包 JS 运行时、三平台打包与 GitHub 发布流水线、迁移前后的性能与体积实测。
- **Out of scope**：
  - pi-web 后端、前端 UI、会话/附件/agent 载入等一切业务逻辑（桌面壳只是它们的宿主）。
  - 代码签名、公证（notarization）、自动更新（本迁移沿用「未签名分发」现状；已知上游缺陷 tauri-apps/tauri#11992 使 sidecar 与 macOS 签名冲突，若未来要签名需另立 spec）。
  - Electron 壳与 Tauri 壳并存（已定为原地替换）。
  - 桌面端新增任何 Electron 时代不存在的原生能力（托盘、通知、菜单等）。
- **Adjacent expectations**：
  - 桌面壳**依赖**自包含产物 `dist/` 已由既有 `build:dist` 流程产出，且其入口恒为产物根的 `dist/server.mjs`；桌面壳不拥有该产物的构建方式。
  - 桌面壳**依赖**后端在就绪后对 HTTP `GET /` 返回任意响应；桌面壳不拥有就绪语义的定义，只复用之。
  - 桌面壳**不注入** agent 配置目录覆盖，从而使会话默认落 `~/.pi/agent`、与 CLI 共享；该目录的结构与语义由 pi-web 后端拥有。
  - 前端经单一访问器读取桌面能力桥；桌面壳**拥有**该桥的注入，**不拥有**前端在桥缺失时的降级渲染决策。

## Requirements

### Requirement 1: 运行模式判定与窗口加载

**Objective:** 作为桌面版用户，我希望应用启动后直接看到可用的 pi-web 界面，以便立即开始会话；作为开发者，我希望开发态能连到热更新的开发服务器。

#### Acceptance Criteria

1. When 应用以未打包形态启动且开发地址环境变量被设为非空值, the 桌面壳 shall 直接加载该开发地址且不拉起本地后端。
2. When 应用以打包形态启动, the 桌面壳 shall 从随包资源目录定位后端入口并拉起本地后端。
3. When 应用以未打包形态启动且未设开发地址, the 桌面壳 shall 从构建产物布局定位后端入口并拉起本地后端。
4. While 本地后端尚未就绪, the 桌面壳 shall 显示加载页而非空白窗口。
5. When 本地后端就绪, the 桌面壳 shall 使窗口加载该后端的本地回环地址。
6. Where 应用运行于 macOS 且所有窗口已关闭, the 桌面壳 shall 在用户经 Dock 重新激活时重开窗口。
7. Where 应用运行于 macOS 之外的平台且所有窗口已关闭, the 桌面壳 shall 退出应用。

### Requirement 2: 后端受监管拉起与就绪判定

**Objective:** 作为桌面版用户，我希望应用能自行找到可用端口并等到后端真正可用后再显示界面，以便不会看到半加载的页面。

#### Acceptance Criteria

1. When 桌面壳准备拉起后端, the 桌面壳 shall 从起始端口起递增探测，选取首个空闲的回环端口。
2. When 后端进程已启动, the 桌面壳 shall 轮询该端口，直至后端对 HTTP `GET /` 返回任意响应即判定为就绪。
3. While 就绪轮询进行中且后端进程已自行退出, the 桌面壳 shall 立即停止轮询并判定为「后端早退」。
4. If 就绪轮询超过 60 秒仍未成功且后端进程仍存活, then the 桌面壳 shall 判定为「就绪超时」。
5. If 从起始端口起的探测范围内无任何空闲端口, then the 桌面壳 shall 判定为「无可用端口」且不启动任何后端进程。
6. When 就绪判定失败, the 桌面壳 shall 先收尾其已拉起的进程，不留孤儿进程。
7. The 桌面壳 shall 使「就绪超时」与「后端早退」互不混淆，即在收尾动作影响进程状态之前完成判别。

### Requirement 3: 启动失败的可读呈现与重试

**Objective:** 作为桌面版用户，当应用起不来时我希望看到能读懂的原因和可操作的选项，以便自行重试或退出，而不是面对一个白窗口。

#### Acceptance Criteria

1. If 后端拉起失败, then the 桌面壳 shall 在窗口内呈现一条区分失败类别（无可用端口 / 后端早退 / 就绪超时）的可读错误说明。
2. When 失败类别为「后端早退」, the 桌面壳 shall 在错误说明中包含后端退出码与其标准错误输出的尾部内容。
3. When 失败类别为「就绪超时」, the 桌面壳 shall 在错误说明中包含所等待的时长。
4. When 后端早退且未产生任何标准错误输出, the 桌面壳 shall 仍呈现一条可读的错误说明。
5. When 用户在错误呈现界面选择重试, the 桌面壳 shall 重新执行完整的拉起流程。
6. When 用户在错误呈现界面选择退出, the 桌面壳 shall 退出应用。

### Requirement 4: 退出收尾进程树

**Objective:** 作为桌面版用户，我希望关闭应用后不残留后台进程、端口被释放，以便下次启动不受影响、机器不被悄悄占用。

#### Acceptance Criteria

1. When 应用退出, the 桌面壳 shall 在真正退出前终止后端进程**及其派生的所有子孙进程**。
2. When 收尾流程被多次触发, the 桌面壳 shall 保持幂等，不重复终止、不抛错、不挂起。
3. When 后端进程在收尾前已自行退出, the 桌面壳 shall 跳过终止动作并正常完成退出。
4. When 收尾流程向后端进程发出优雅终止请求后超过宽限期仍未退出, the 桌面壳 shall 强制终止该进程树。
5. When 应用完成退出, the 桌面壳 shall 使先前占用的本地端口在有限时间内可被重新占用。

### Requirement 5: 随包 JS 运行时与「无系统 Node 可用」保证

**Objective:** 作为在干净机器上安装桌面版的用户，我希望即使系统从未安装过 Node 也能正常使用，以便无需先配置开发环境。

#### Acceptance Criteria

1. The 桌面壳 shall 随安装包携带一份 JS 运行时二进制，其目标平台与架构与该安装包一致。
2. When 桌面壳拉起后端, the 桌面壳 shall 使用随包携带的 JS 运行时执行后端入口，而非系统路径上的任何运行时。
3. When 桌面壳拉起后端, the 桌面壳 shall 经环境变量把随包 JS 运行时的**绝对路径**下达给后端，使后端派生的 agent 子进程复用同一运行时。
4. While 系统可执行文件搜索路径中不存在任何 Node 可执行文件, when 用户发起一次真实会话, the 桌面壳 shall 使该会话正常完成（含 agent 子进程启动与流式回复）。
5. The 桌面壳 shall 使会话数据落于用户默认 agent 目录，与命令行版本共享，即不注入任何 agent 目录覆盖。

### Requirement 6: 原生目录选择能力

**Objective:** 作为桌面版用户，我希望能用系统原生的文件夹选择器挑选 agent 源目录，以便不必手工输入路径。

#### Acceptance Criteria

1. Where 应用运行于桌面壳内, the 桌面壳 shall 向渲染层暴露一个受控的能力桥，其形状与既有前端契约一致（含只读标识、平台标识、目录选择方法）。
2. When 渲染层调用目录选择方法, the 桌面壳 shall 打开系统原生的「选择文件夹」对话框。
3. When 用户在对话框中选定一个目录, the 桌面壳 shall 仅回传该目录的绝对路径字符串。
4. If 用户取消对话框或未选定任何目录, then the 桌面壳 shall 回传「无结果」而非抛出错误。
5. If 打开对话框的过程发生异常, then the 桌面壳 shall 降级为「无结果」语义、记录诊断信息，且不使调用方收到拒绝的 Promise。
6. The 桌面壳 shall 不经该桥回传目录内容、文件列表或任何文件系统元数据。
7. Where 应用运行于普通浏览器（非桌面壳）, the 前端 shall 读取不到该能力桥，从而不渲染「浏览文件夹」入口，其余功能不受影响。

### Requirement 7: 外链处理

**Objective:** 作为桌面版用户，我希望点击外部链接时在我的默认浏览器里打开，以便不把应用窗口导航离开；同时我不希望应用被诱导打开危险的链接类型。

#### Acceptance Criteria

1. When 页面请求打开一个外部 `http` 或 `https` 地址, the 桌面壳 shall 在系统默认浏览器中打开该地址，且不使应用窗口导航离开。
2. If 页面请求打开的地址指向本地回环主机, then the 桌面壳 shall 拒绝该请求。
3. If 页面请求打开的地址使用 `http`/`https` 之外的协议, then the 桌面壳 shall 拒绝该请求。
4. If 页面请求打开的地址无法被解析为合法地址, then the 桌面壳 shall 拒绝该请求且不抛出错误。

### Requirement 8: 安全基线

**Objective:** 作为安全负责人，我希望桌面壳的攻击面不因本次技术栈更换而扩大，以便迁移不引入新的风险。

#### Acceptance Criteria

1. The 桌面壳 shall 使后端仅监听本地回环地址，不监听任何对外网络接口。
2. The 桌面壳 shall 使后端监听的端口在每次启动时按可用性选取，而非固定公开端口。
3. The 桌面壳 shall 仅向渲染层授予其明确需要的宿主能力，不授予通用文件系统访问或任意命令执行能力。
4. Where 渲染层被授予调用宿主能力的权限, the 桌面壳 shall 把该授权的来源限定为本地回环地址。
5. The 桌面壳 shall 不因迁移而新增任何 Electron 时代未暴露给渲染层的宿主能力。

### Requirement 9: GitHub 发布流水线

**Objective:** 作为维护者，我希望打一个版本标签就能自动产出三平台安装包并附加到 GitHub Release，以便发布过程可重复、可审计。

#### Acceptance Criteria

1. When 版本标签被推送, the 发布流水线 shall 为 macOS（Apple Silicon 与 Intel 两种架构）、Windows、Linux 分别产出安装包。
2. The 发布流水线 shall 按**目标架构**而非仅按操作系统展开构建矩阵，使 macOS 的两种架构各自产出一份安装包。
3. When 某个矩阵分支开始构建, the 发布流水线 shall 为该目标架构准备与之匹配的随包 JS 运行时二进制。
4. When 随包 JS 运行时二进制被获取, the 发布流水线 shall 校验其完整性（校验和），且在校验失败时使该构建失败而非产出未经校验的安装包。
5. If 任一平台的构建失败, then the 发布流水线 shall 不阻止其余平台继续构建，且失败信息可在流水线日志中定位。
6. When 所有矩阵分支完成且触发源为版本标签, the 发布流水线 shall 把各平台安装包附加到该标签对应的 GitHub Release。
7. When 流水线由手动方式触发, the 发布流水线 shall 产出可下载的工作流产物，且不附加到任何 GitHub Release。
8. The 发布流水线 shall 在产出安装包后、附加到 Release 之前，于至少一个平台上对**已打包的应用**执行一次真实会话冒烟验证。
9. The 发布流水线 shall 使 Linux 安装包的可执行文件名不含会被安装包格式拒绝的字符。

### Requirement 10: 迁移后的验证等价物

**Objective:** 作为维护者，我希望迁移后仍有与迁移前同等强度的端到端证据，以便相信行为确实等价、而不是只在纸面上等价。

#### Acceptance Criteria

1. The 验证套件 shall 覆盖迁移前四条端到端验证各自断言的可观察事实：未打包真实会话、已打包应用真实会话、无系统 Node 下真实会话与退出后端口释放、原生目录选择回传路径。
2. When 「无系统 Node 下真实会话」的验证执行, the 验证套件 shall 实际从可执行文件搜索路径中剥除所有 Node 可执行文件后再启动应用，且该项验证不得以任何形式降级为「假定可用」。
3. When 「已打包应用真实会话」的验证执行, the 验证套件 shall 启动**实际打包产物**而非未打包构建，以捕获仅在打包形态下暴露的资源缺失类回归。
4. Where 目标平台不存在可驱动应用内网页视图的自动化工具, the 验证套件 shall 以黑盒方式验证：应用拉起的本地回环端点可用、真实会话经该端点完成、相关进程与端口在退出后被释放。
5. The 验证套件 shall 对进程编排逻辑（端口选取、就绪判定、失败分类、进程树收尾、随包运行时路径解析、外链放行判定）提供不依赖图形界面的自动化测试。
6. When 验证套件执行完毕, the 验证套件 shall 以实际运行输出作为通过与否的证据，而非以代码审阅结论替代。

### Requirement 11: 迁移动机的实测验收

**Objective:** 作为决策者，我希望用实测数据确认这次迁移真的换来了更低的内存、更快的启动和更小的包体，以便判断它是否值得，而不是接受一个理论上的承诺。

#### Acceptance Criteria

1. When 迁移实现完成, the 验收报告 shall 给出迁移前后**空闲常驻内存**的实测对比，测量方法与采样时机在两侧一致。
2. When 迁移实现完成, the 验收报告 shall 给出迁移前后**冷启动至界面可交互**耗时的实测对比，测量方法在两侧一致。
3. When 迁移实现完成, the 验收报告 shall 给出迁移前后**安装包体积**的实测对比，且分别列出三平台（macOS 两架构、Windows、Linux）的数值。
4. The 验收报告 shall 在包体对比中显式计入随包 JS 运行时二进制所贡献的体积。
5. If 包体积的净收益不显著, then the 验收报告 shall 显式记录该结论并交由决策者重新裁定本次迁移是否继续，而非默认继续。
6. The 验收报告 shall 不以「新方案理论上更轻」一类论证替代任何一项实测数值。
