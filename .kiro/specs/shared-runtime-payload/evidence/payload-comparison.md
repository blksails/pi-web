# 净收益的实测验收报告

_spec `shared-runtime-payload` 任务 9.2 — Req 10.2/10.3/10.4、12.1–12.8_

本报告以**实测数值**裁定这次「压缩载荷 + 共享运行时」的改造是否值得。
不以「压缩后理论上更小」一类论证替代任何一项数值（Req 12.8）。

## 测量条件

| 项 | 值 |
|---|---|
| 平台 | macOS 24.6.0，Apple Silicon（`aarch64-apple-darwin`） |
| **改造前** | 提交 `98e7e94`（含第一层剪枝），在临时 worktree 中以**同一工具链**重建 `.app` 后实测 |
| **改造后** | 本分支 |
| 测量脚本 | `scripts/measure-payload-baseline.mjs --repeat 5`（两侧同脚本、同口径） |
| 汇总脚本 | `scripts/payload-summary.mjs`（由上面两份原始 JSON + `du -sm` 生成 summary，**不手写数字**） |
| 裁定脚本 | `scripts/payload-verdict.mjs`（阈值以断言实现，可复跑，不达标退出码 1） |
| 原始数据 | `evidence/measure-before.{json,log}`、`evidence/measure-after.{json,log}`、`evidence/measure-summary.json` |

**可复现**：`measure-payload-baseline --repeat 5` → `payload-summary` → `payload-verdict`。
裁定所依据的每一个数字都能由仓库中的脚本重新产出；每一轮冷启动的原始值都落盘。

**口径定义**（两侧完全一致）：

- **下载体积**：`hdiutil create -format UDZO` 产出的 dmg 字节数——与用户实际下载的形态一致。
  ⚠ dmg **本身就是压缩格式**，故「85MB 的 dist 压成 9.4MB」并不等价于下载体积降到九分之一。
- **磁盘占用**：一律 `du -sm`（块占用）。不与 `npm pack` 报告的字节和混用，二者不可比。
- **冷启动**：从 `spawn` 到**后端首次响应 `GET /`**。稳态取 **5 次运行的中位数**——
  单次测量的噪声可达 ±400ms，足以淹没本改造的真实影响。

## 结果

### 下载体积

| | 改造前 | 改造后 | 变化 |
|---|---:|---:|---:|
| `.app` | 177 MB | **101 MB** | −42.9% |
| **dmg（用户实际下载）** | 81.4 MB | **47.6 MB** | **−41.5%** |

### 磁盘占用（三种安装场景）

| 场景 | 改造前 | 改造后 | 变化 |
|---|---:|---:|---:|
| 仅装桌面版 | 177 MB | 190 MB | **+13 MB** |
| 仅装 CLI | 86 MB | 99 MB | **+13 MB** |
| **两者都装**（共享一份运行时） | 263 MB | **200 MB** | **−63 MB** |

**拆解**（Req 12.3：显式计入「载荷 + 解包副本各存一份」的重复存储）：

| 组成 | 改造前 | 改造后 |
|---|---:|---:|
| `.app` 内的产物 | 未压缩 `dist/` 85 MB | 压缩载荷 **9.4 MB** |
| npm 包内的产物 | 未压缩 `dist/` 85 MB | 压缩载荷 **9.4 MB** |
| 共享运行时目录 | — | **89 MB**（一份，两产品共用） |
| 随包 node（不属本 spec） | 86 MB | 86 MB（未动） |

**单产品用户为此付出 +13 MB**：载荷（9.4 MB）在磁盘上与解包副本并存，且解包树因
dereference 比 `dist/` 大 3–4 MB（11 个符号链接展开，`packages/*` 复制 489 个文件）。
这是本方案的固有取舍，不是可以优化掉的实现缺陷。

### 启动耗时

| | 改造前 | 改造后 | 变化 |
|---|---:|---:|---:|
| **稳态**冷启动（5 次中位数） | 1101 ms | 1137 ms | **+36 ms** |
| 5 次原始值 | 2769 / 1101 / 1101 / 1096 / 1161 | 1157 / 1421 / 1137 / 1097 / 1089 | — |
| **首次**启动（含真实解包） | — | **6974 ms** | 一次性 |

稳态路径只读 `payload.json` 与 `.ok` 两个小文件，不读取 9.4 MB 载荷、不重算摘要（Req 10.1）。

⚠ **关于测量方法的更正**：本报告的早期版本曾以**单次**测量给出「+1 ms」，且那组「5 次中位数」
无法由仓库中的任何脚本复现——独立复查据此判定 REJECT，理由成立。现已把 `--repeat N` 做进
`measure-payload-baseline.mjs`，每轮原始值随 `.log` 一并落盘，中位数由 `payload-summary.mjs`
计算。诚实的稳态增量是 **+36 ms**，不是 +1 ms。

离群值确实存在且不对称：改造前第 1 轮 2769 ms（页缓存全冷），改造后第 2 轮 1421 ms。
两侧的最小值分别是 1096 / 1089 ms，几乎相同。中位数正是为抵御这类噪声而取；
本报告不采用任何单次测量值。

## 阈值裁定（Req 12.4–12.7、10.2）

```
✓ 安装包下载体积降幅     41.5%（81.4 → 47.6 MB）     要求 ≥ 25%      [Req 12.4]
✓ 仅装桌面版的磁盘增量    +13 MB                      要求 ≤ +20 MB   [Req 12.5]
✓ 仅装 CLI 的磁盘增量     +13 MB                      要求 ≤ +20 MB   [Req 12.5]
✓ 两者都装的磁盘净节省    63 MB（263 → 200 MB）       要求 ≥ 50 MB    [Req 12.6]
✓ 稳态冷启动增量（中位数） +36 ms                      要求 ≤ +200 ms  [Req 10.2]

裁定：四条阈值全部达标，净收益成立，改造继续。
```

裁定由 `node scripts/payload-verdict.mjs` 产出，**不达标即以退出码 1 停止**（Req 12.7）。
该脚本已做反向验证：把「仅装桌面版」的磁盘占用改成 +23 MB 后，它确实输出
「净收益不成立…应停止并交回决策者」并返回退出码 1。

## 收益归属

- **下载体积 −41.5%** 是无条件收益，所有用户都拿到。
- **磁盘 −63 MB** 只在「CLI 与桌面版都装」时兑现。单装用户**多付 13 MB**。
  若绝大多数用户只装其一，本改造在磁盘维度上是净亏的——这一点不因阈值通过而消失，
  只是被「下载体积」这一更重要的维度盖过。

## 尚未覆盖的平台

本报告仅覆盖 **macOS arm64**。Windows 与 Linux 的下载体积、三场景磁盘、冷启动
需在对应平台或 CI 上实测后补入，**不得据本报告推断其他平台的收益**（见 tasks.md 的盲区表）。
