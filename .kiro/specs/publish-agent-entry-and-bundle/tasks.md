# Implementation Plan

> 执行前提:改动全在 pi-web 单仓;**pi-clouds 零改动**。所有新增失败面必须落在 `compile()` 内(早于 upload/register/setChannel 三段外部写),以保证失败不消耗版本号。
>
> **并行性说明**:任务 3、4、5 逻辑上互相独立,但**都改同一个文件** `server/cli/publish/manifest-compiler.ts`,并发执行会产生 merge 冲突,故一律不标 `(P)`。真正可并行的是分处不同文件的 6、8。
>
> **夹具纪律**:所有编译期用例一律基于**临时目录中的拷贝**构造包夹具,**不得原地修改或删除 `examples/` 下的任何真实示例**(仅任务 8 例外,它是有意为该示例新增清单文件)。

  - 新增可选的通用文件白名单字段,接受 glob 字符串数组
  - 在 web 字段组内新增自动探测开关,缺省为开启
  - 观察点:对既有的 3 个示例清单(`canvas-component-watermark` / `module-settings-agent` / `plugin-code-review-agent`)解析结果不变;缺少 kind 的清单解析报错
  - _Requirements: 2.3, 3.6, 4.1, 4.3_
  - _Boundary: packages/protocol 清单契约_

- [x] 2. 缺陷回归取证(先于实现,用于复现 #28)
  - 在拉起真实 in-proc registry 的端到端测试文件中,新增一个类型为 agent、含入口文件与路由子目录的包的完整发布→安装用例;包夹具在临时目录中构造
  - **此时必须失败**,且失败原因必须是清单缺少入口声明(`VALIDATION: manifest.entry must be an object`)——复现不出该原因说明夹具没走到校验,用例无效
  - 将该失败原因原文记录在用例注释中,作为 #28 的复现锚点

  - 观察点:运行该用例,输出中出现上述失败原因原文
  - _Requirements: 6.1_
  - _Boundary: test/publish/publish-install-e2e.test.ts_

- [x] 3. 编译层:入口判定与打包通道(#28 核心)

- [x] 3.1 扩展编译产物与错误类型
  - 为编译产物增加入口引用字段(仅 agent 类型产出)与非阻断警告集合字段
  - 新增五个编译期错误码:类型未声明、入口不存在、入口覆盖文件缺失、入口越出包目录、有 webext 源但无产物
  - 类型未声明的错误需携带可选取值集合,供呈现层列出
  - 观察点:类型定义通过编译检查;既有调用方无需改动即可通过类型检查
  - _Requirements: 4.2_

- [x] 3.2 接入运行时入口探测约定
  - 从 server 包的公开导出引入既有的入口探测能力与其覆盖缺失异常,不复制其判定逻辑
  - 仅在包类型为 agent 时执行探测;非 agent 类型不产出入口声明,即使包内存在符合约定的入口文件
  - 将覆盖文件缺失、无任何入口两种情形分别映射为对应错误码,并在提示中给出补救方式
  - 判定探测结果是否位于包目录之内,越界则报错拒绝(服务端会拒绝包外路径,不前置拦截就会烧版本号)
  - 观察点:在临时目录构造一个含 `index.ts` 的 agent 包夹具,编译得到入口指向该文件;移除夹具中的入口文件后编译报「入口不存在」,且全程未发生任何网络调用
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6_

- [x] 3.3 将入口纳入发布产物并写入签名清单
  - 把探测到的入口文件加入待打包文件集合,并计算其完整性摘要
  - 在签名清单构造中输出入口声明;仅当入口存在时输出该字段
  - 确认入口与资源字段命中同一文件时不产生重复打包(文件集合天然去重),且不影响签名规范化结果
  - 观察点:任务 2 的回归用例转为通过,登记后的版本状态为可用状态
  - _Requirements: 1.1, 2.1, 6.1_

- [x] 4. 编译层:附属文件打包通道(#29)

- [x] 4.1 实现通用文件白名单的展开
  - 展开清单声明的文件白名单,将命中文件加入待打包文件集合,但**不加入**需完整性核验的引用集合
  - 任一声明模式零命中时,沿用既有的「声明路径缺失」错误语义,并指出未命中的模式
  - 观察点:夹具声明 `routes/**` 后,该目录下文件出现在打包文件清单中,且未出现在核验引用中;把模式改成不存在的路径则编译失败
  - _Requirements: 2.3, 2.4, 2.5_

- [x] 4.2 将包元数据文件纳入发布产物
  - 当包根存在包元数据文件时无条件加入待打包文件集合
  - 该文件不进入核验引用集合
  - 说明必要性:入口覆盖声明存放于该文件,不打包会导致安装后运行期探测不到覆盖,与发布期判定产生错位
  - 观察点:发布产物文件清单中包含该文件;安装后的目录中该文件存在
  - _Requirements: 2.2_

- [x] 5. 编译层:webext 产物通道(#29)

- [x] 5.1 实现「显式优先 + 约定探测兜底」
  - 保留显式声明分支的既有行为完全不变
  - 未显式声明时,探测约定产物目录下的产物清单文件,存在则自动纳入整棵产物树并在签名清单中输出 web 扩展声明
  - 约定路径需与运行时解析所用的默认值保持同一取值
  - 观察点:对「已有产物但未声明」的夹具编译,签名清单中出现 web 扩展声明且产物树进入打包文件清单
  - _Requirements: 3.1, 3.2_

- [x] 5.2 实现有源无产物的硬失败
  - 检测到 webext 源文件存在但对应产物不存在时,以专用错误码终止编译
  - 错误信息中必须给出应先执行的构建命令,而非仅陈述产物缺失
  - 不得静默跳过 webext——这正是生产上面板失效的直接成因
  - 观察点:对「有源无产物」的夹具编译立即失败,且提示中含可直接复制执行的构建命令
  - _Requirements: 3.3, 3.4_

- [x] 5.3 实现陈旧产物警告与自动探测开关
  - 比对产物清单文件与 webext 源文件的修改时间,产物较旧时产出警告,但不阻断发布
  - 清单显式关闭自动探测时跳过约定路径探测,且不因产物缺失而失败
  - 观察点:把夹具产物时间戳改早后编译,发布仍成功且警告集合非空;关闭开关后即使存在产物,签名清单中也无 web 扩展声明
  - _Requirements: 3.5, 3.6_

- [x] 6. 呈现层:错误文案与警告输出 (P)
