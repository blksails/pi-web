## Implementation Notes

- **任务 4.1（2026-07-21）**：`readAt` 曾只对 `ENOENT` 返回 `{}`，而 `list`/`exists` 都把 `ENOTDIR` 视为「不存在」——同一实例对同一个键给出互斥答案，违反 Req 2.1。复核抓出并正向对齐。**教训**：涉及「不存在」判定的方法必须共享**唯一**口径，逐方法各写各的 errno 分支必然漂移。
- **任务 4.1（2026-07-21）**：packages/server 的实测基线是 **234 文件面 / 1939 用例**（不是先前记的 104）。后续任务按此比对。
- **任务 4.1 复核升级 → 契约勘误⑦⑧**：两条跨实现分歧由**契约**收口而非由实现各自决定——
  ⑦ 大小写敏感是键空间语义，大小写不敏感载体上**保证不了**，实现须声明限制、套件不得用仅大小写不同的键对验证其它维度；
  ⑧ 值与分组不可同址（Req 1.7/1.8），落地在 4.2、套件固化在 4.4。
  **教训**：实现层遇到「载体做不到」时，正确动作是上报契约，不是在实现里悄悄定一个只有自己遵守的规矩。
- **平台红线**：Windows 设备名 / NTFS ADS 在键空间层面**合法**，能否落盘由各实现声明；**跨平台真机验证前，任何文档与报告都不得宣称 Windows 可用**。
- **任务 4.2（2026-07-21）**：反向同址校验一度把「路径在载体上是目录」当成「存在冲突值键」。二者只在 `delete` 之前等价——`delete` 只删值不删父目录，此后层级载体上残留的**空目录**在扁平 KV 上根本不存在。修法引入删目录动作后，最大风险变成数据丢失，故 `removeEmptyDirTree` **只调 `readdir`/`rmdir`，绝不 `unlink`、绝不 `fs.rm({recursive})`**：判据即使写错，内核也会用 `ENOTEMPTY` 挡住。**教训**：当修复需要引入破坏性原语时，先让原语本身在物理上做不到最坏的事，再谈判据对不对。
- **任务 4.2（2026-07-21）**：errno 兜底在层级 FS 上**原理上不够**——① 勘误⑧ 要求错误指名冲突键，errno 给不出；② 空分组必须写入**成功**，而它与非空分组在 errno 层面都是 `EISDIR`。故必须主动探测；errno 只能作并发窗口的兜底，且**兜底后必须重新探测再分类**，否则真实 IO 故障会被误报成「宿主键写错了」。契约初稿「层级 FS 免费」已据此改正。
- **任务 4.2（2026-07-21）· 方法**：「这条断言有判别力吗」的最终答案是**变异测试**——把实现临时改回错误写法，捕获红，再还原。本任务用它把一句论证变成了证据（`expected 'key' to be 'io'`）。成本约两分钟，比争论便宜。
- **任务 4.3（2026-07-21）· ★方法论上的关键一课**：一条测试**声称**自己是守卫，不等于它是。本任务有一条用例自述「将来 guard 被摘掉时留下可读判据」，复核者用变异体实测：**guard 完全摘掉（M0）时它仍是绿的**，M6/M7 下亦全绿——零杀伤力，且与 4.2 既有用例是同入口同形状的纯重复。**一条伪装成守卫的恒真断言比普通恒真断言更坏**，因为它让人以为那个方向有人看着。判据只有一个：拿变异体杀它，杀不动就是没用。
- **任务 4.3（2026-07-21）**：`NaN` 上限曾**静默失效**——`size > NaN` 恒为 `false`，任意大的值都写得进去。env 路径对同样的非法值是 fail-fast 的，两条入口口径分裂就是缺陷。校验落点选在 `createLocalWorkspaceNamespace`（上限进入比较的**唯一咽喉**），变异体 M7（把 guard 上移到装配层）证实那样会漏掉直接调用工厂的路径。
- **任务 4.3（2026-07-21）**：实施者给落点的**理由**曾有一半是错的（称一致性套件走命名空间工厂，实际 `ConformanceTarget.workspace` 是 `Workspace` 类型，走的是装配入口）。结论对不代表论证对——**注释里断言一个假事实比不写更糟**，未来读者会依赖它做判断。
- **任务 4.3（2026-07-21）**：`WorkspaceConfigError.envName` → `source`。改名的**期限**是该类经 barrel 成为对外公开面之前（6.1/6.2）；一旦 pi-clouds 跨仓消费，成本就从 3 行变成跨仓破坏性变更。**教训**：共享类型上的语义污染有到期日，越界与否要按到期日算，不能只按当前任务边界算。
- **design.md 曾自相矛盾**：`LocalWorkspace` 的 Responsibilities 要求它承担根解析，接口块却把两根定为**必填**——必填则该职责无处可落。已按「职责 + 契约 §3.5」改为可选并补 `env`/`cwd` 注入项。
- **任务 4.4（2026-07-21）· ★★覆盖纪律（本 spec 最重要的一条）**：`delete`/`exists` 把键校验写在 `try` 内，`WorkspaceKeyError` 被 errno 兜底重包成 `WorkspaceIoError`（判别码 `key → io`），**经 4.1/4.2/4.3 三轮独立复核全绿而未被发现**。复核者用变异体实证：还原该缺陷后 `local-workspace.test.ts`(36) + `local-workspace-assembly.test.ts`(23) **依旧全绿**，只有一致性套件红 28 条。**教训**：单测覆盖「校验函数本体」+「方法正常路径」= 端口层**错误判别码零覆盖**。凡契约按 `code` 分流的方法，**每个方法**都必须有一条「非法输入 → 期望 code」的用例；靠校验函数的单测替代它，是覆盖率上的幻觉。
- **任务 4.4（2026-07-21）**：这是「套件即契约的可执行部分」第一次兑现——套件照契约写、实现照契约写，二者**首次对接**就抓出三轮复核都没抓到的缺陷（32 failed/62）。**两份独立的契约解读相互检验，比同一份解读被检查三遍更有效。**
- **任务 4.4（2026-07-21）· 声明不是豁免**：`ConformanceTarget.caseSensitiveKeys` 声明 `false` 后，用例**不跳过**而是**反向断言**（两键确实互为别名、后写者胜出、列举只回一条），谎报同样红。豁免式的声明等于让实现自证清白——凡「实现可声明某条不适用」的机制，都必须设计成谎报会被抓出，否则它就是一个后门。已写入契约勘误⑪。

### 编排层教训（多 agent 并发，2026-07-21 第 5 组）

第 5 组三个 `(P)` 任务并行执行，暴露出一批**与代码质量无关、纯属编排缺陷**的问题。它们不影响交付物，但每一条都真实浪费了上下文或差点造成数据丢失。根因几乎全在派单模板。

- **★ 权威文档不是变异靶标**。契约文档被某复核者备份为变异基线，还原时整体覆盖，**静默销毁**了期间写入的勘误⑫——而 `git status` 干净，零信号。要验「文档↔代码双向失配」，须复制到自己的 scratchpad 改副本，或让解析函数接受路径参数；**绝不原地改再还原**。同时：**写完勘误立即 commit**，未提交的活文档等于一直待在可被静默回滚的窗口里。
- **★ scratchpad 是全会话共享的**，按会话 UUID 分配而非每 agent 一份。所有人都会挑 `bak/`、`backup/` 这种通用名，故备份互相覆写**不是偶发而是必然**（一位复核者的回滚依据就这么没了）。派单模板须**指名** `scratchpad/<agent-name>/`，只说「用独立子目录」不够。
- **★ 变异窗口须广播**。一个正当的变异测试会让并发的旁观者读到中间态，进而把它误判成「实施者在偷改代码」。开跑前后各广播一次；被复核方在窗口内保持只读。**还原确认要用逐行核对或 sha 校验，不要只依赖 diff 备份**——备份本身可能已被别人覆写。
- **★ 撞见别人的活变异体，不要顺手还原**。那会在对方跑到一半时抽掉靶子，与销毁勘误是同一类事故。正确做法是报告并把还原责任留给窗口持有者。
- **vitest 在并发负载下会「假绿」**：报 `Tests no tests` 而**退出码 0**。一次没跑到任何用例的运行，在脚本里看起来完全等同于成功。判据必须是「输出里有真实用例数」，不能只看退出码。根因经判别实验定位为 monorepo 级 chokidar 监听压垮 vite-node 主线程（已提交的旧测试同样失败 → 与改动无关；仓外探针同等负载 10.3s 通过 → 排除机器饱和）。
- **`TS2578 Unused '@ts-expect-error'` 作为「负向断言失效」的信号，只在其余错误为零时可信**。一条上游 `TS2307`（模块解析不到）会把被断言的类型全退化成 error 类型，连带让整批指令变「未使用」——实测 1 条带出 9 条。判据须写成：**在其余错误为零的前提下**仍剩任何一条 = 该断言什么也没守。
- **子代理的纯文本输出到不了主控**，必须走 SendMessage。本轮四个子代理踩中，其中一个连丢两次报告。派单模板须明写这一条。
- **★ 主控的自省**：本轮我两次看到互相矛盾的观测就往「有人虚报/越权」推，两次都错，代价都落在被怀疑方（各花一轮上下文自证）。在多 agent 共享工作树的环境里，**更平凡的解释总是「有人正在做一件正当的事，只是你不知道」**。先穷举正当解释，再考虑失信。
- **★ 并发下报告纪律：只报自己那一侧**。本轮反复出现的一类错误可归结为一句——**把「我这边为真」讲成「客观为真」**。三种形态：① 在窗口通告里写一个没真的算过的哈希（为让消息「显得完整」）；② 自己停止写入后就断言「整个目录没有任何写者」（实际另一个 agent 刚开窗）；③ 各自的 `typecheck` 观测互相矛盾，于是怀疑有人虚报（实际是三个不同瞬间的中间态）。可确证的只有「我不再写」，写者是否为零得由编排层判定。**收敛这类矛盾要问「谁在何时跑」，而不是「谁对谁错」。**
- **★ 任务 5.1（2026-07-21）· 三轮才把一个机制说准**：实施者跑变异体 N2 推翻了自己写的注释解释（「(b) 补的是 EPC 可被中间变量绕过的漏」）→ 我把它的结论写进契约，但把定语落错了地方（写成「EPC 在 async 经 `Promise<T>` 包装的位置根本不触发」）→ 复核者用两轮隔离探针证伪**我们俩都错**：`async f(): Promise<T> { return {…多余属性} }` **会**触发 `TS2353`，真正的分界是**返回类型来自显式注解（EPC 触发）还是上下文推断（不触发）**，四种写法 EPC 只挡两种、`never` 四种全挡。**教训**：① 契约是给别的仓库读的，里面一句不实的机制说明会误导两端实现者，而它**不影响任何测试是否通过**——只有主动去探才会发现；② 更精确的机制反而**更强地**支持保留那两条断言（价值不是「补漏」而是「覆盖 EPC 够不到的一半形态」），故必须把「为何不冗余」的**正确理由**写下来，否则后人会拿「杀伤力相同」推出「删一个」；③ 自我更正的痕迹要留在文档里（「初稿此处曾写…是错的」），让后来者知道这个结论被实测推翻过，不会想当然地简化回去。
- **★ 变异体不得因「同族」而省略复跑**。任务 5.1 有现成反例：M3 与 M7 都动 `models` 字段，杀伤位置却不同（M3 只死在反向断言，M7 双向都死）。同族直觉在这个文件上已被证伪过一次。复跑成本约 12 秒/个，省不得。
- **★ 任务 5.3（2026-07-21）· 「保守处理」何时是安全的**：`registry.ts` 的转译只认 `code === "key"`，其余原样上抛——这个保守方向本身不构成盲区，**当且仅当另有守卫盯着那个方向**：若将来 `validateWorkspaceKey` 改抛别的判别码，`registry.test.ts` 那 8 条真实非法 id 用例会**集体转红**，漂移不可能静默通过。**没有守卫的「保守」只是把问题藏起来。** 判据：说「我宁可保守」时，要能指出哪条断言会在保守失效时报警。
- **任务 5.3（2026-07-21）· 错误分类的通用形态**：`catch` 里无区分地把异常转成领域错误码，会把**实现故障伪装成调用方的输入错误**。这与 4.2 的「按 errno 盲目分类」是同一种病：**用「错误在哪发生」替代「错误是什么」来定性**。修法是先按稳定判别码判别、命中才转译、其余原样上抛。锁它的断言要用一个**非该错误类实例、只带判别码**的对象——这样同时钉住了勘误① 的「按 `code` 不按 `instanceof`」纪律（变异体把判据改回 `instanceof` 即转红）。
- **任务 5.3（2026-07-21）· 不要为可测性动契约面**：曾考虑给工厂加「注入校验器」参数以便打桩，被否决。理由：那等于**让测试需求反向修改一个刚冻结的对外公开面**——该参数会被两端看见、被迫理解、甚至被误用，而它存在的唯一原因是想 stub 一个纯函数。`vi.mock` 的代价局限在测试文件内，且用 `importOriginal` 收窄到单个导出后连维护耦合都消失。
- **★ 任务 5.1（2026-07-21）· 把安全边界从文档义务升级为编译期保证**：Req 5.3「附件授予仅在带会话标识时出现」原本只能靠文档约束——契约签名 `load(sessionId?)` 的返回类型与实参无关，越权签发无人可挡。**重载形态挡不住**（复核者写探针实证：TS 不逐条校验重载实现签名，`return { attachments }` 编译通过），那是 4.3「伪装成守卫的恒真断言」的类型层翻版，差点被立进契约。改双方法后各自有被诚实校验的返回类型，实证可挡（勘误⑫a）。**判据**：一个「机械保证」若挡不住它声称要挡的那个动作，比没有保证更坏——它让人以为那个方向有人看着。
- **任务 5.1（2026-07-21）**：契约 §4.1 的 `attachments.endpoint` 与实现的 `baseUrl` 不一致，且**改名从未被记为有意为之**。失败形态：pi-clouds 照契约发 `{ endpoint }`，pi-web 读 `baseUrl` 得 `undefined`，跨仓静默不匹配。这类「文档与实现各写各的字段名」是本 spec 存在的理由本身，已随勘误⑫b 收口。
- **「同族」是关于变异体来源的直觉，杀伤力取决于断言覆盖的方向——两者不是一回事。** 这是「同族所以不复跑」为何不成立的准确表述。
- **★★ 任务 5.2（2026-07-21）· 为什么自查替代不了独立复核（本 spec 三次栽在同一处的结构性原因）**：实施者为 M1–M4 各造了变异体、全部被杀，看起来judgment 严谨；但复核者造的 **M9b**（缺失表态抛错**前**对已表态者急切调用工厂）**存活**——那条自述守卫「抛错前零副作用」的断言，spy 绑在**未表态**的对象上（任何实现都不会调它），唯一已表态者的工厂是内联的、没有变量可断言。实施者自己的总结最准：**「自己造的变异体只能验证自己想到的失效模式」**——M1–M4 全打在他设想过的方向，缺陷恰在他没想到的方向。**这不是勤勉程度问题，是盲区的结构性问题。独立复核的价值不在「再看一遍」，在于换一个盲区。**
- **任务 5.2（2026-07-21）· 命名会成为缺陷的掩护**：那条无牙守卫里的变量叫 `used`，实质却是「**未**被表态」的那个。名字与实质相反，使「断言绑错了对象」在阅读时不显眼。整改一并改名 `used → unstated`。
- **★ 禁止一个方向，就要同时开一条接缝**：本轮立了「权威文档禁止原地变异」的禁令，但 `readFrozenIdsFromContract()` 硬编码文档路径、对照物不可注入——**禁令因此把「文档改→用例转红」这个方向直接判死**（该方向此前靠原地变异实证过，禁令后不可复验）。复核者的表述值得记：协议禁止原地变异，就必须同时提供副本注入的接缝。修法是把路径提成带默认值的形参。
- **★★ 还原不是撤销，是一次写入**（勘误⑫ 被静默销毁的真正机制，比「多 agent 共用 scratchpad」更根本）：
  - **机制**：校验和只能证明「它变回了我备份时的样子」，证明不了「那就是它此刻应有的样子」。还原的正确性取决于「备份时刻之后有没有别人写过」，而**校验和不含时间语义**，对这个问题完全沉默。
  - **反直觉处**：**校验越严格，误判越笃定**。全绿的 sha 会把「我做对了」钉死，反而**抑制**了「我是不是该先看看 `git log` / `git diff`」这个动作。
  - **可执行判据**：动共享文件前先问「**这个文件有没有第二个写者**」，而不是「我的副本对不对」。有第二写者时，唯一安全的还原是 `git checkout <file>`（以版本库为仲裁者，它带时间语义），**永远不是**用自己的副本覆盖。
- **转述他人观测必须带时刻**。主控曾把一位实施者「M1 变异体还活着」的瞬时观测当作当前态转发，而那其实是复核者某轮运行中的快照——这是「把『我这边为真』讲成『客观为真』」的变体：**把「某人某时看到的」讲成「现在是」**。
- **★★ 两条盲区教训是同构的，可以合成一句**：「我的备份不含别人的写入」与「我的变异体不含我没想到的失效模式」——**凡是拿『自己手里的那份』当全集的，都会以全绿的形式失败**。这也正是独立复核不可替代的结构性理由：不是第二双更勤勉的眼睛，是**第二个搜索空间**。盲区按定义不在自己的搜索空间里，勤勉补不了。
- **命名可以把误解编码进标识符**（任务 5.2 复核者的分析）：那条无牙守卫里的变量叫 `used`，而本模块的表态类型里 `kind: "use"` 恰恰就是「沿用」——于是 `const used = routeFactory("a")` 读起来天然像「表态为 use 的那个」，**而这正是导致缺陷的那个错误信念本身**。原名不是中性的坏名字，它让「拿它断言工厂未被调用」看上去理所当然。改名 `unstated` 后，该断言的恒真性**当场可见**。**判据**：当一个名字与所在领域的术语撞义时，它会替错误辩护。

### 取证方法的教训（第 6 组，2026-07-22）

- **★★ 用一次侥幸正确的结论去追认一个不成立的取证方法** —— 本轮两例并列，同一种病：
  - 主控在 4.1 用 `cmd 2>&1 | tail -3; echo "EXIT=$?"` 取退出码，那是 **`tail` 的**不是 `tsc` 的；结论碰巧对（后续多人独立复验过），于是**错误的方法被结果背书**。
  - 复核者用「6 分 30 秒静默」推断变异窗口已交还并开跑，事后证明**结论正确**（对方全扫恰在那前结束）——**正因为结论正确，这个错误方法差点被这轮结果追认**。若对方的 P1 那时开跑，两人就在同一文件上互抽靶子了。
  - **判据**：结论对不代表方法对。方法的正确性要独立于这一次的结果来判断。
- **★ 管道使 `$?` 失真**：`cmd | tail; echo $?` 取到的是 `tail` 的退出码。
  **⚠ 修正（实测）：`${PIPESTATUS[0]}` 是 bash 的，本会话 shell 是 zsh —— zsh 用 `$pipestatus`（小写、下标从 1）。** 实测 `(exit 3) | cat` 后：`${PIPESTATUS[0]}` = **空串**，`${pipestatus[1]}` = `3`。**空串在条件判断里等同于成功，故这个错法沉默失败。**
  ——本条最初写进 Notes 时用的正是 `${PIPESTATUS[0]}`，是错的；由实施者指出、主控实测证伪。**「记录一条纪律」不等于「这条纪律是对的」，同样要验。**
  可靠做法：**不用管道**（`cmd > /dev/null 2>&1; echo $?`），或 `set -o pipefail`，或 zsh 下用 `$pipestatus`。
  且**退出码不能单独作判据**——vitest 并发下会 rc=0 而无用例数，必须同时确认输出里有真实用例数。**7.1 的四面回归全是管道形式，开跑前必须定死这条。**
- **★ 报哈希须带算法与时刻**（`sha256@00:44`）。`shasum` 默认 sha1（40 位），`shasum -a 256` 是 64 位；两者不等时第一反应是「文件被改过」而非「我在比两种算法」——这是「哈希不带时刻就成假事实」的**算法版**，且**沉默失败**。
- **★ 观测工具本身的缺陷会被当成被观测对象的状态**：一位复核者报「`index.ts` 被删」，实为**自己的探测脚本用了相对路径**——文件从未消失。主控随后在提交 Notes 时用相对路径 `cat >>`，因 cwd 停在子包而**内容根本没写进去**，同一课当场重演。判据：跨目录操作一律用**绝对路径**；报「文件消失」这类强断言前先绝对路径复验。
- **判断「这个变异体是不是我的」要用内容，不要用时间**：mtime 只能说明「有人动过」，而「这个符号我从没碰过」是**排他**证据。本轮据此正确排除了一次误认。
- **变异窗口协议补充**（多 agent 并发）：① 开窗广播须声明**预计时长**与**运行形态**（前台/后台脚本）；② 超 5 分钟须发**心跳**；③ **「agent 空闲」≠「窗口关闭」**——空闲是会被推送的事件，「盘上有没有活靶」不是，唯一权威是 **CLOSE 广播**；④ **untracked 文件的窗口没有版本库兜底**（`git checkout` 救不回来），还原完全依赖持有者自己的副本。
- **撞见别人的活变异体不要顺手还原**——会在对方跑到一半时抽掉靶子。报告并把还原责任留给窗口持有者。本轮双方各正确执行过一次。
- **守卫可能不是被人删掉的，是被工具「优化」掉的**：`import type { … }` 若无人消费，编辑器的 Organize Imports 会摘掉它——而它正是类型面守卫的**本体**。摘除后**没有任何东西报警**（`noUnusedLocals` 未开、名集断言对类型沉默、`tsc` 照样 rc=0）。故留一个把它们消费掉的元组作**锚**。⚠ 但要写准：本仓**没有** eslint/biome/oxlint/prettier，风险来自编辑器手动动作而非 CI——**「锚」不是「守卫本身」**。
- **★ 任务 6.1（2026-07-22）· barrel 守卫必须分两层，且两层的强制点在不同地方**：
  - **值导出漏项** → 只有**运行期名集断言**抓得到（`tsc` 对半个 barrel 照样 rc=0）；
  - **类型导出漏项** → 只有 **`tsc`** 抓得到（运行期用例**全绿沉默**，实测删一个 `type` 导出 → `3 passed`）；
  - 类型层的**真守卫是 `import type { … }` 语句本身**（TS 对具名类型导入急性解析，与是否被使用无关，8/8 全杀），把它归功给「用掉这些类型的元组别名」是**错的**——双向实测：只删元组、出口完整 → `tsc` rc=0，元组的**边际贡献恰好为 0**。
  - 元组仍要留，但它是**「防止守卫被摘除」的锚**，不是守卫本身。
- **★ 注释正确性只有主动核对能兜底**（实施者自述）：**代码正确性有测试兜底，注释正确性没有**。本任务同一段头部注释里被抓出**三处**不实——依赖清单（`node:crypto` 漏写，自查发现）、安全边界误标（复核抓出）、「各文件」以偏概全（复核抓出）。**没有任何测试会因为注释写错而变红。**
- **「安全边界」这类强词要对着契约核**：`resolveWorkspaceKeyPath` 被说成「本端口封起来的安全边界」，但契约 §3.2 第 3 条把安全边界指派给**键校验**，而键校验本体恰恰是**导出**的、`resolveWorkspaceKeyPath` 第一件事也是调它。误标会让读者得出「安全边界没被导出」这个假结论。**不导出它的正确理由是层次错位（fs 载体专有概念放进载体无关的端口面）+ 绕过端口原子写/权限/上限/同址校验的近路。**
- **★ 引文与哈希不带时刻，读者会把旧态当现态**——本轮出现了**镜像的两次**：① 实施者给了过期哈希，主控拿它比对现态而起疑；② 复核意见基于整改前的旧快照，把已改好的一处报成「仍未处理」。**双向都会发生**，故凡引用文件内容或哈希，一律标注时刻（`sha256@00:49`）。
- **转述一句话不会让它变成事实**：实施者在引用主控的措辞（「organize-imports / lint autofix 会移除」）时把它写进注释，而该措辞有一半经不起 grep（本仓无 lint）。**该 grep 还得 grep**——转述不豁免核实义务，哪怕来源是上级。
- **★★ 上级的肯定会让一句不实的自述获得追认**（任务 6.2，实施者主动自曝）：它在开窗广播里称「已单列一条断言钉住 capability 运行期导出数为 0」，**后半句当时并未写**；主控在回信里**恰恰肯定了那个设计**并要求把推理写进注释——于是一个不存在的东西被追认了。它事后主动纠正并补上。**机制**：一句自述被上级引用、赞许、并要求"完善"之后，就从「待核实的声明」变成了「已确立的事实」，**再也不会有人回头去核它**。**判据**：① 报告里描述"我做了什么"必须与盘上状态逐条对齐，设计意图与已交付要分开写；② 上级复述你的自述时，若与实际不符，**当场纠正**——沉默等于确认；③ 主控侧：肯定一个设计之前先看盘上有没有，别据摘要赞许。
- **★ 守卫的前提本身也要有守卫**（任务 6.2）：一条「遍历模块运行期导出逐个比对」的断言，其**前提**是「该模块有运行期导出」。`capability` 是纯类型模块（运行期导出数 0），那一行会变成**遍历空集、永远为绿**。前提悄悄变化时，断言不会报错——它只会**变得没有意义**。修法不是删掉它，而是**把前提本身做成可转红的断言**（钉死「运行期导出数恒为 0」），将来该模块长出运行期导出时转红，**强制**把它加回遍历表，而不是让新符号躺在守卫之外。
- **★ 新交付的东西天然无人看守**：`capability` 漏接主入口是**完全静默**的——运行期守卫结构上不可能有杀伤力（导出数为 0），`tsc` 也不会红，**因为没有任何既有代码引用它**。「有消费者」是大多数守卫的隐含前提，而端口面本就先于消费者存在。这类模块的覆盖 100% 落在类型层。
- **★ vitest 假绿有两种形态，判据只能是「输出里的真实用例数」**：① 收集阶段静默无果 → `Tests no tests` + **rc=0**（多人多次复现）；② 收集阶段 worker 超时报错 → `Tests no tests` + **rc=1**，输出含 `Errors 1 error`（`[vitest-worker]: Timeout calling "fetch" with "/@vite/env"`，一次观测）。
  **操作性结论**：「rc=0 就是通过」不可靠，**「rc≠0 就是失败」同样不可靠**——两个方向都只能用真实用例数兜。
  ⚠ **归因要写准**：主控曾据②那一次观测推广成「假绿退出码不稳定/随机」，被复核者收窄——**一次样本不是规律**。耗时实测：`Duration 454s`（transform 203s / collect 377s / **tests 0.45s**），开销全在 transform+collect，与用例无关，与「monorepo 级监听压垮 vite-node」的既有归因一致。
- **`trap` 的路径必须在编写时就绝对化**——它执行时的 cwd 不由你控制。「跨目录用绝对路径」对**立即执行**的命令够用，对**延迟执行**的清理钩子不够：在 `cd` 之前写的 `trap`，会在 `cd` 之后触发。本轮一位复核者据此踩中，`trap` 里的 `cp` 报 `No such file or directory`、**还原没执行**（40 秒内以绝对路径补救）。**这类失误特别隐蔽：`trap` 失败时你正在退出，没人在看输出。**
