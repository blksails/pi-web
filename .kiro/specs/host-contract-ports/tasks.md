## Implementation Notes

- **任务 4.1（2026-07-21）**：`readAt` 曾只对 `ENOENT` 返回 `{}`，而 `list`/`exists` 都把 `ENOTDIR` 视为「不存在」——同一实例对同一个键给出互斥答案，违反 Req 2.1。复核抓出并正向对齐。**教训**：涉及「不存在」判定的方法必须共享**唯一**口径，逐方法各写各的 errno 分支必然漂移。
- **任务 4.1（2026-07-21）**：packages/server 的实测基线是 **234 文件面 / 1939 用例**（不是先前记的 104）。后续任务按此比对。
- **任务 4.1 复核升级 → 契约勘误⑦⑧**：两条跨实现分歧由**契约**收口而非由实现各自决定——
  ⑦ 大小写敏感是键空间语义，大小写不敏感载体上**保证不了**，实现须声明限制、套件不得用仅大小写不同的键对验证其它维度；
  ⑧ 值与分组不可同址（Req 1.7/1.8），落地在 4.2、套件固化在 4.4。
  **教训**：实现层遇到「载体做不到」时，正确动作是上报契约，不是在实现里悄悄定一个只有自己遵守的规矩。
- **平台红线**：Windows 设备名 / NTFS ADS 在键空间层面**合法**，能否落盘由各实现声明；**跨平台真机验证前，任何文档与报告都不得宣称 Windows 可用**。
- **任务 4.2（2026-07-21）**：反向同址校验一度把「路径在载体上是目录」当成「存在冲突值键」。二者只在 `delete` 之前等价——`delete` 只删值不删父目录，此后层级载体上残留的**空目录**在扁平 KV 上根本不存在。修法引入删目录动作后，最大风险变成数据丢失，故 `removeEmptyDirTree` **只调 `readdir`/`rmdir`，绝不 `unlink`、绝不 `fs.rm({recursive})`**：判据即使写错，内核也会用 `ENOTEMPTY` 挡住。**教训**：当修复需要引入破坏性原语时，先让原语本身在物理上做不到最坏的事，再谈判据对不对。
- **任务 4.2（2026-07-21）**：errno 兜底在层级 FS 上**原理上不够**——① 勘误⑧ 要求错误指名冲突键，errno 给不出；② 空分组必须写入**成功**，而它与非空分组在 errno 层面都是 `EISDIR`。故必须主动探测；errno 只能作并发窗口的兜底，且**兜底后必须重新探测再分类**，否则真实 IO 故障会被误报成「宿主键写错了」。契约初稿「层级 FS 免费」已据此改正。
- **任务 4.2（2026-07-21）· 方法**：「这条断言有判别力吗」的最终答案是**变异测试**——把实现临时改回错误写法，捕获红，再还原。本任务用它把一句论证变成了证据（`expected 'key' to be 'io'`）。成本约两分钟，比争论便宜。
- **任务 4.3（2026-07-21）· ★方法论上的关键一课**：一条测试**声称**自己是守卫，不等于它是。本任务有一条用例自述「将来 guard 被摘掉时留下可读判据」，复核者用变异体实测：**guard 完全摘掉（M0）时它仍是绿的**，M6/M7 下亦全绿——零杀伤力，且与 4.2 既有用例是同入口同形状的纯重复。**一条伪装成守卫的恒真断言比普通恒真断言更坏**，因为它让人以为那个方向有人看着。判据只有一个：拿变异体杀它，杀不动就是没用。
- **任务 4.3（2026-07-21）**：`NaN` 上限曾**静默失效**——`size > NaN` 恒为 `false`，任意大的值都写得进去。env 路径对同样的非法值是 fail-fast 的，两条入口口径分裂就是缺陷。校验落点选在 `createLocalWorkspaceNamespace`（上限进入比较的**唯一咽喉**），变异体 M7（把 guard 上移到装配层）证实那样会漏掉直接调用工厂的路径。
- **任务 4.3（2026-07-21）**：实施者给落点的**理由**曾有一半是错的（称一致性套件走命名空间工厂，实际 `ConformanceTarget.workspace` 是 `Workspace` 类型，走的是装配入口）。结论对不代表论证对——**注释里断言一个假事实比不写更糟**，未来读者会依赖它做判断。
- **任务 4.3（2026-07-21）**：`WorkspaceConfigError.envName` → `source`。改名的**期限**是该类经 barrel 成为对外公开面之前（6.1/6.2）；一旦 pi-clouds 跨仓消费，成本就从 3 行变成跨仓破坏性变更。**教训**：共享类型上的语义污染有到期日，越界与否要按到期日算，不能只按当前任务边界算。
- **design.md 曾自相矛盾**：`LocalWorkspace` 的 Responsibilities 要求它承担根解析，接口块却把两根定为**必填**——必填则该职责无处可落。已按「职责 + 契约 §3.5」改为可选并补 `env`/`cwd` 注入项。
- **任务 4.4（2026-07-21）· ★★覆盖纪律（本 spec 最重要的一条）**：`delete`/`exists` 把键校验写在 `try` 内，`WorkspaceKeyError` 被 errno 兜底重包成 `WorkspaceIoError`（判别码 `key → io`），**经 4.1/4.2/4.3 三轮独立复核全绿而未被发现**。复核者用变异体实证：还原该缺陷后 `local-workspace.test.ts`(36) + `local-workspace-assembly.test.ts`(23) **依旧全绿**，只有一致性套件红 28 条。**教训**：单测覆盖「校验函数本体」+「方法正常路径」= 端口层**错误判别码零覆盖**。凡契约按 `code` 分流的方法，**每个方法**都必须有一条「非法输入 → 期望 code」的用例；靠校验函数的单测替代它，是覆盖率上的幻觉。
- **任务 4.4（2026-07-21）**：这是「套件即契约的可执行部分」第一次兑现——套件照契约写、实现照契约写，二者**首次对接**就抓出三轮复核都没抓到的缺陷（32 failed/62）。**两份独立的契约解读相互检验，比同一份解读被检查三遍更有效。**
- **任务 4.4（2026-07-21）· 声明不是豁免**：`ConformanceTarget.caseSensitiveKeys` 声明 `false` 后，用例**不跳过**而是**反向断言**（两键确实互为别名、后写者胜出、列举只回一条），谎报同样红。豁免式的声明等于让实现自证清白——凡「实现可声明某条不适用」的机制，都必须设计成谎报会被抓出，否则它就是一个后门。已写入契约勘误⑪。

### 编排层教训（多 agent 并发，2026-07-21 第 5 组）

第 5 组三个 `(P)` 任务并行执行，暴露出一批**与代码质量无关、纯属编排缺陷**的问题。它们不影响交付物，但每一条都真实浪费了上下文或差点造成数据丢失。根因几乎全在派单模板。

- **★ 权威文档不是变异靶标**。契约文档被某复核者备份为变异基线，还原时整体覆盖，**静默销毁**了期间写入的勘误⑫——而 `git status` 干净，零信号。要验「文档↔代码双向失配」，须复制到自己的 scratchpad 改副本，或让解析函数接受路径参数；**绝不原地改再还原**。同时：**写完勘误立即 commit**，未提交的活文档等于一直待在可被静默回滚的窗口里。
- **★ scratchpad 是全会话共享的**，按会话 UUID 分配而非每 agent 一份。所有人都会挑 `bak/`、`backup/` 这种通用名，故备份互相覆写**不是偶发而是必然**（一位复核者的回滚依据就这么没了）。派单模板须**指名** `scratchpad/<agent-name>/`，只说「用独立子目录」不够。
- **★ 变异窗口须广播**。一个正当的变异测试会让并发的旁观者读到中间态，进而把它误判成「实施者在偷改代码」。开跑前后各广播一次；被复核方在窗口内保持只读。**还原确认要用逐行核对或 sha 校验，不要只依赖 diff 备份**——备份本身可能已被别人覆写。
- **★ 撞见别人的活变异体，不要顺手还原**。那会在对方跑到一半时抽掉靶子，与销毁勘误是同一类事故。正确做法是报告并把还原责任留给窗口持有者。
- **vitest 在并发负载下会「假绿」**：报 `Tests no tests` 而**退出码 0**。一次没跑到任何用例的运行，在脚本里看起来完全等同于成功。判据必须是「输出里有真实用例数」，不能只看退出码。根因经判别实验定位为 monorepo 级 chokidar 监听压垮 vite-node 主线程（已提交的旧测试同样失败 → 与改动无关；仓外探针同等负载 10.3s 通过 → 排除机器饱和）。
- **`TS2578 Unused '@ts-expect-error'` 作为「负向断言失效」的信号，只在其余错误为零时可信**。一条上游 `TS2307`（模块解析不到）会把被断言的类型全退化成 error 类型，连带让整批指令变「未使用」——实测 1 条带出 9 条。判据须写成：**在其余错误为零的前提下**仍剩任何一条 = 该断言什么也没守。
- **子代理的纯文本输出到不了主控**，必须走 SendMessage。本轮四个子代理踩中，其中一个连丢两次报告。派单模板须明写这一条。
- **★ 主控的自省**：本轮我两次看到互相矛盾的观测就往「有人虚报/越权」推，两次都错，代价都落在被怀疑方（各花一轮上下文自证）。在多 agent 共享工作树的环境里，**更平凡的解释总是「有人正在做一件正当的事，只是你不知道」**。先穷举正当解释，再考虑失信。
- **★ 并发下报告纪律：只报自己那一侧**。本轮反复出现的一类错误可归结为一句——**把「我这边为真」讲成「客观为真」**。三种形态：① 在窗口通告里写一个没真的算过的哈希（为让消息「显得完整」）；② 自己停止写入后就断言「整个目录没有任何写者」（实际另一个 agent 刚开窗）；③ 各自的 `typecheck` 观测互相矛盾，于是怀疑有人虚报（实际是三个不同瞬间的中间态）。可确证的只有「我不再写」，写者是否为零得由编排层判定。**收敛这类矛盾要问「谁在何时跑」，而不是「谁对谁错」。**
- **★ 任务 5.1（2026-07-21）· 三轮才把一个机制说准**：实施者跑变异体 N2 推翻了自己写的注释解释（「(b) 补的是 EPC 可被中间变量绕过的漏」）→ 我把它的结论写进契约，但把定语落错了地方（写成「EPC 在 async 经 `Promise<T>` 包装的位置根本不触发」）→ 复核者用两轮隔离探针证伪**我们俩都错**：`async f(): Promise<T> { return {…多余属性} }` **会**触发 `TS2353`，真正的分界是**返回类型来自显式注解（EPC 触发）还是上下文推断（不触发）**，四种写法 EPC 只挡两种、`never` 四种全挡。**教训**：① 契约是给别的仓库读的，里面一句不实的机制说明会误导两端实现者，而它**不影响任何测试是否通过**——只有主动去探才会发现；② 更精确的机制反而**更强地**支持保留那两条断言（价值不是「补漏」而是「覆盖 EPC 够不到的一半形态」），故必须把「为何不冗余」的**正确理由**写下来，否则后人会拿「杀伤力相同」推出「删一个」；③ 自我更正的痕迹要留在文档里（「初稿此处曾写…是错的」），让后来者知道这个结论被实测推翻过，不会想当然地简化回去。
- **★ 变异体不得因「同族」而省略复跑**。任务 5.1 有现成反例：M3 与 M7 都动 `models` 字段，杀伤位置却不同（M3 只死在反向断言，M7 双向都死）。同族直觉在这个文件上已被证伪过一次。复跑成本约 12 秒/个，省不得。
