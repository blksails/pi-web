# Implementation Plan

- [x] 1. Foundation:会话创建契约与显式会话标识
- [x] 1.1 扩展会话创建请求以支持恢复标识
  - 在会话创建请求 DTO 中新增可选的 `resumeId` 字段,语义为"存在即恢复,缺失即新建"
  - 保持既有新建请求向后兼容(无 `resumeId` 时行为不变)
  - 完成态:协议包类型检查通过,既有请求构造不受影响,`resumeId` 可被序列化/解析
  - _Requirements: 3.1_
- [x] 1.2 会话管理器支持显式会话标识
  - 允许会话创建时传入显式会话标识,显式标识优先于随机生成;缺省仍随机生成
  - 完成态:传入显式标识时新建会话以该标识登记,既有随机标识路径单测仍通过
  - _Requirements: 2.2_

- [x] 2. Core:agent 子进程持久化与标识对齐(真实模式)
- [x] 2.1 runner 按会话标识 open-or-create(对齐 pi CLI 语义)
  - runner 解析启动参数(会话标识、模型、源元数据);缺省参数保持既有行为
  - 模仿 pi CLI(main.js:255-261):按会话标识查已有会话文件,存在则打开加载历史,否则以该标识新建,使持久化文件标识与主进程标识一致
  - 完成态:以新标识启动后持久化产物标识等于该标识;以已存在标识启动时 agent 加载到既有历史
  - _Requirements: 2.2, 3.3_
  - _Boundary: runner 恢复参数_
- [x] 2.2 runner 写入创建元数据并保持写入容错
  - 新建会话时写入一条创建元数据(agent source、工作目录、模型)随会话持久化;恢复时不重复写入
  - 存储写入为旁路、失败不中断对话(沿用既有 best-effort 镜像)
  - 完成态:新建会话后存储中存在该创建元数据条目;注入写入错误时对话流不中断
  - _Requirements: 1.1, 1.4, 1.5_
  - _Boundary: runner 恢复参数_
- [x] 2.3 验证创建元数据在文件后端可读回(SPIKE 固化)
  - 真实 runner 写入创建元数据后,主进程经存储读接口按标识读回,确认条目类型与字段一致
  - 字段不一致时调整读写映射使之对齐
  - 完成态:存在一个可复跑的验证(node-e2e),证明文件后端写入的创建元数据能被主进程读回且字段齐全
  - _Requirements: 1.3, 1.4_
  - _Depends: 2.2_

- [x] 3. Core:主进程冷会话恢复编排
- [x] 3.1 主进程按标识读取恢复元数据
  - 主进程按配置后端惰性建立存储读取器,按会话标识读取权威工作目录与创建元数据(source、model)
  - 会话不存在或读取失败时返回"无元数据"而非抛出
  - 完成态:对已持久化会话返回 `{source, cwd, model}`;对不存在标识返回空,且不抛错到调用方
  - _Requirements: 1.4, 3.1_
  - _Boundary: ResumeMetaLoader_
- [x] 3.2 会话创建端点新增恢复分支
  - 携带 `resumeId` 的创建请求走恢复:内存已活跃则幂等返回;否则据恢复元数据解析源并以显式标识创建活跃会话;元数据缺失返回可识别的"会话不存在"
  - 完成态:对冷会话返回 201 且会话变为活跃可续聊;对不存在标识返回 404;对活跃会话重复请求不重建
  - _Requirements: 3.1, 3.2, 3.4_
  - _Depends: 1.1, 1.2, 3.1_
- [x] 3.3 主进程装配恢复读取器与按模式拼接会话参数
  - 主进程按存储配置注入恢复读取器;构造通道时据源模式追加会话参数:custom 模式传会话标识/源元数据/模型,cli 模式传会话标识/模型(pi 原生,不传源元数据);恢复时工作目录取自持久化头部而非默认值
  - 完成态:custom 与 cli 两模式下经恢复分支创建的会话,其 agent 以正确工作目录与标识启动并加载历史
  - _Requirements: 2.2, 3.1, 3.3_
  - _Depends: 3.1, 3.2, 2.1_

- [x] 4. Core:stub agent 持久化与恢复(支撑双后端 e2e)
- [x] 4.1 stub 进程获得存储访问与会话上下文
  - stub 启动时注入运行时加载器以复用既有存储实现(失败回退内联极简存储),并经环境传入会话标识、恢复标志、工作目录、源、模型
  - 完成态:stub 进程可在文件与 sqlite 两后端建立存储句柄而不崩溃
  - _Requirements: 1.1, 1.2, 1.3_
  - _Boundary: stub agent_
- [x] 4.2 stub 持久化对话并支持按标识恢复
  - 新建时写入会话头部与创建元数据;每次对话顺序持久化用户消息与最终回复(写后可读);恢复时按标识读回历史重建消息列表;新增历史消息查询返回完整历史
  - 完成态:stub 对话后存储含对应消息条目;以恢复标志重启后历史消息查询返回既有对话
  - _Requirements: 1.1, 1.4, 3.3_
  - _Boundary: stub agent_
  - _Depends: 4.1_
