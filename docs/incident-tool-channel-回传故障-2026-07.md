# 事件报告：工具结果回传通道间歇性故障

> **反馈对象**：Claude Code / Harness 工具层团队
> **报告来源**：pi-web 开发会话（AI 助手自述）
> **日期**：2026-07-18（跨 2 个 session）
> **严重度**：HIGH
> **任务结局**：已安全完成（提交/合并/推送均经权威校验）

---

## 1. 执行摘要

在一次常规的「提交 → 合并 → 推送代码」任务中，harness 的**工具结果回传（tool-result）通道**出现严重的间歇性故障，导致任务被拉长至跨两个会话才完成。

**一句话定性**：故障位于**「工具结果回传」路径，而非「命令下发」路径**——命令在服务端确实执行了，只是执行结果没能回到模型侧。

绝大多数 `Bash` / `Read` / `ToolSearch` 调用返回**空输出**（连 `echo hi` 也收不到回显），仅有随机、短暂的恢复窗口。更严重的是，通道抖动期间工具调用被**重复投递**，使后台脚本并发执行多次，在 `.gitignore` 中留下重复 3 次的脏数据——这是一个已经落地的真实副作用。任务最终依靠「带锁+安全门的脚本 + 日志持久化 + git 权威状态校验」的降级策略安全完成。

## 3. 观测到的症状

> 按对「正确性 / 安全性」的威胁程度排序。

### 症状 A — 结果回传大面积返回空 · `HIGH`
- 最简命令（`echo hi`、`echo alive`）也返回空，排除命令本身出错。
- 故障跨 `Bash` / `Read` / `ToolSearch` 多类工具，指向**通用的结果回传层**，而非某个具体工具。
- 命令实际已执行（后台脚本产物可证），仅结果丢失。

### 症状 B — 间歇性 · 随机短暂恢复窗口 · `MEDIUM`
- 故障非持续——期间零星调用能成功回传，经验值约**每 5–8 次成功 1 次**。
- 恢复窗口出现时机随机、持续短暂，无法预测。

### 症状 C — 输出错位 / 串扰 · `CRITICAL`
- 一次工具调用返回的内容，竟是**上一条 `cat` 命令的旧输出**，而非当前命令结果。
- 危险性：若模型轻信便会误判当前操作状态。
- 暗示**请求-响应关联（correlation / ID 匹配）存在错乱**。

### 症状 D — 工具调用被重复投递 · `CRITICAL`
- `run_in_background` 启动的脚本被**并发执行多次**：日志出现大量交错、重复的执行记录（多个 `STEP0` / `STEP6` / `RESULT=SUCCESS`，时间戳成对出现）。
- **实际后果**：脚本中「`grep -q || append`」式的幂等检查在并发下失效，`.gitignore` 被追加了**重复 3 次的 `.pnpm-store/` 与 2 次的 `.agents/`**。
- 这是本次唯一造成落地副作用的症状，表明重试机制**缺乏去重 / 幂等保护**。

### 症状 E — 多行输出被截断 · `MEDIUM`
- 较长的多行输出常只回传前几行即中断，需将结果压成极简单行或分片读取才能取全。

## 4. 实证危害与修复时间线

> 症状 D 从「日志噪声」升级为「真实数据污染」，并在下一会话被诊断、修复、验证——这是本报告的核心因果链。

| 阶段 | 事件 |
|---|---|
| **Session 1 · 抖动期** | 后台脚本 `run_in_background` 启动，因通道抖动被**重复投递并发执行**。git 内容寻址保证 commit 幂等，最终提交/合并/推送结果正确——但副作用潜伏。 |
| **遗留副作用** | 脚本的幂等检查在并发下失效，`.gitignore` 被追加**重复 3 次的 `.pnpm-store/`、2 次的 `.agents/`**。 |
| **Session 2 · 诊断** | 下一会话核对 `.gitignore` 时发现重复块，定位为上一会话重复投递的遗留脏数据。 |
| **Session 2 · 修复 + 加固** | 脚本改用 **`mkdir` 原子锁**防重复投递，并顺手去重 `.gitignore`。本次日志**干净无交错**——锁保护验证有效；结果经 `ls-remote` 外部权威确认（`origin/main` = `76d5820`）。 |

## 5. 影响评估

- **效率**：约 5 分钟的常规任务被拉长至跨两个会话，大量调用消耗在空转重试上。
- **安全风险（已规避）**：commit / merge / push 均不可逆或外向。无反馈时贸然操作有把缺陷代码合入 main 的风险。
- **数据污染（已发生）**：重复投递使 `.gitignore` 出现重复条目，需后续清理——已在 Session 2 修复。
- **用户体验**：用户反复催促「继续」，而模型在无反馈下无法安全推进，造成明显卡顿与挫败。

---

## 6. 缓解措施（验证有效的降级模式）

> 当通道表现出「偶发性丢输出」时，应尽早切换到这套模式，而非在实时交互里空转。

1. **后台 / 服务端脚本执行** — 把操作封装成自包含脚本，使执行不依赖实时 stdout 回传。
2. **脚本内建安全门** — `set -e` 串联 + 关键闸门：测试通过才提交、`--ff-only` 才合并、远程地址不符不覆盖、暂存集不含忽略路径。把正确性/安全性从「模型实时判断」下沉到「脚本自身门禁」。
3. **原子锁防重复投递** — `mkdir` 锁确保脚本只跑一次，直接针对症状 D，Session 2 验证有效。
4. **结果持久化到文件** — 每步写日志文件，再反复重试读取，趁恢复窗口取回结果。
5. **幂等设计** — 使重复投递不破坏状态（git 内容寻址、push 同一 commit 无害）。
6. **只信权威状态** — 用 `git rev-parse` / `log` / `ls-remote` 校验内容寻址的真实状态，不轻信可能错位的日志表象。
7. **规避错位与截断** — 命令结果先写文件再读；判断压成极简单行（如 `PUSHED_OK` / `NOT_YET`）提高短窗口命中率。

## 7. 给项目组的建议（按优先级）

| 优先级 | 建议 |
|---|---|
| **P1** | **排查结果回传路径丢包（症状 A/B）**：增加回传确认 / 重传机制与超时可观测日志，定位 tool-result 在何种条件下被丢弃或超时。 |
| **P1** | **修复响应-请求关联错乱（症状 C）**：严格校验每个 tool-result 与其调用的 ID；错配时**报错而非静默返回旧内容**——这是正确性红线。 |
| **P1** | **工具调用去重 / 幂等保护（症状 D）**：重试机制应携带幂等键，避免同一 `run_in_background` 被重复投递。本次已有实证危害。 |
| **P2** | **空返回应显式报错**：区分「命令真的无输出」与「回传失败」，后者向模型抛明确错误，否则模型无法判断该重试还是接受。 |
| **P2** | **排查多行输出截断（症状 E）**：确认是否存在回传缓冲 / 分片上限，长输出应完整或显式标记截断。 |
| **P3** | **提供通道健康探针**：给模型一个轻量的「通道是否健康」自检信号，便于在故障期主动切换到降级模式。 |
