# 产品文档审核规范（REVIEW-CHECKLIST）

> 本文件是 `docs/product/` 下每篇章节的定稿门槛。每个写作 subagent 产出后，reviewer subagent 必须按本规范**两轮、五维度**逐条核对并修订；两轮各维度全部 PASS 才算定稿。

## 两轮 review 的定位

- **第 1 轮 · 技术准确性审核（对照代码/spec）** — 重点抓「写的和仓库实际是否一致」。必须对照 `README.md`、`.kiro/steering/`、对应 `.kiro/specs/` 和真实源码，逐条核对路径、API、环境变量、命令、协议字段。
- **第 2 轮 · 可读性与完整性审核（对照读者）** — 重点抓「目标读者能不能照着做成」。检查结构、范例可运行性、步骤连贯、交叉链接、术语一致。

## 五个维度 checklist（每篇套用）

| 维度 | 审核要点 |
| --- | --- |
| **准确性** | 文件路径/行号、包名（`@blksails/*`）、env 变量名（如 `PI_CODING_AGENT_DIR` 而非 `PI_AGENT_DIR`）、API 端点、命令、默认值是否与代码一致；不臆造未实现的能力（区分「已实现」与「路线图」）。 |
| **完整性** | 该主题的关键子项无遗漏；前置条件、限制/约束、失败路径都覆盖；与 README/steering 不冲突。 |
| **可读性** | 一句话定位开头、表格/图示得当、段落不冗长；中文表述自然，技术名词保留原文；标题层级合理。 |
| **范例** | 至少一个**可运行/可复制**的代码或命令范例；范例与正文一致；尽量来自真实 `examples/` 而非虚构。 |
| **步骤清晰** | 操作类内容用有序步骤；每步可独立验证；给出预期结果与常见报错对策（链到 [23 FAQ](./23-troubleshooting-faq.md)）。 |

## 五条补充规范

1. **证据要求** — 凡声称「跑得通」的命令，reviewer 需实际执行或对照已有 e2e 证据，禁止凭记忆判定（沿用项目 `kiro-verify-completion` 原则）。
2. **交叉链接校验** — 第 2 轮专门检查 `[xx](./xx.md)` 链接目标存在且锚点正确。
3. **一致性基线** — 术语、目录编号、表头风格以 `README.md` 与 `00/01/02` 三篇为模板基线，后续章节不得偏离。
4. **范围纪律** — 路线图能力必须显式标注「规划中/未实现」，不能与当前可用能力混写。
5. **通过门槛** — 两轮各维度全部 PASS 才算定稿；任一维度 FAIL 需返修并复审该维度。

## 评分记录格式

reviewer 每轮对五维度给出 `PASS / FAIL / NA`，FAIL 必附原因与已做修订，并复审至 PASS：

```
round: 1 | 2
accuracy:     PASS | FAIL(原因/修订)
completeness: PASS | FAIL(原因/修订)
readability:  PASS | FAIL(原因/修订)
examples:     PASS | FAIL(原因/修订)
steps:        PASS | FAIL | NA(非操作类)
remainingConcerns: [...]   # 仍需人工确认的点
```
